Auto-log bank SMS via iPhone Shortcut

An iPhone Automation watches incoming bank texts for credit, debit, or transfer, extracts the amount, and posts it to a separate inbox queue in Firestore — never your full ledger doc. This app watches that queue live and routes each item automatically, checked in this order: mentions one of your other own banks → Internal Transfer (regardless of "credited"/"debited" wording); otherwise credits matching your salary keywords → Income; any other credit or transfer → Cashflow / Transfers; everything else → Expenses. Nothing is ever overwritten. Build one Automation per bank account (one for SBI, one for Kotak, one for Axis, etc.) — each just hardcodes which account it belongs to, so every SMS lands against the right balance automatically, whichever account it's from.

Fallback account — only used if a Shortcut doesn't send an account name 1 — POST URL for Shortcuts "Get Contents of URL" (same for every bank's Automation) 2 — Request Body (JSON) — set "account" to that Automation's exact account name, fill amount/type/raw from Shortcuts variables 3 — Build one Automation per bank account
  1. Shortcuts app → Automation → + → Personal Automation → Message → set sender to that bank's SMS ID (leave "Contains" blank if the sender is bank-only) → Run Immediately.
  2. Add action Get Text from Input (the incoming message).
  3. Add Match Text to detect the keyword: pattern credit(ed)?|debit(ed)?|transfer(red)? — if no match, stop the Shortcut here.
  4. Add Match Text again to pull the amount: pattern (?:Rs\.?|INR|₹)\s?[0-9,]+(?:\.[0-9]{1,2})?, then strip commas/currency symbols so it's a plain number.
  5. Set a Text variable for type: transfer if the message contains "transfer" or "transferred"; otherwise credit if it contains "credited"; otherwise debit. (The app decides Income vs. Cashflow/Transfer for you from there — a "credit" only becomes Income if it matches your salary keywords above; otherwise it's filed as Cashflow/Transfer along with every "transfer".)
  6. Set a Text variable for account: type it exactly as one of the names shown in the box above (e.g. SBI Savings a/c) — this is the one thing that differs between each bank's Automation.
  7. Add Get Contents of URL: Method POST → URL from box 1 above → Headers Content-Type: application/json → Request Body JSON, matching the template in box 2 with your amount/type/account/message variables inserted.
  8. Repeat steps 1–7 for each remaining bank account, changing only the sender and the account name.
4 — Firestore rules (paste into Firestore → Rules)
match /databases/{db}/documents { match /ledgerApp/{syncKey} { allow read, write: if true; // tighten to your syncKey pattern } match /ledgerApp/{syncKey}/inbox/{docId} { allow create: if request.resource.data.keys().hasAll(['raw','amount','type','createdAt']); allow read, delete: if true; } }
These rules are intentionally minimal-friction for a personal setup — an obscure Sync Key is your main protection. If you verified your Identity above (recommended), use this tighter version instead, which locks the main document to only your signed-in phone number:
match /databases/{db}/documents { match /ledgerApp/{syncKey} { allow read, write: if request.auth != null && request.auth.uid == syncKey; } match /ledgerApp/{syncKey}/inbox/{docId} { allow create: if request.resource.data.keys().hasAll(['raw','amount','type','createdAt']); allow read, delete: if true; } }
Note the inbox subcollection stays open either way — the iPhone Shortcut posts to it directly over plain HTTP with no Firebase sign-in, so it can't satisfy an auth != null check. Its real protection is that your Sync Key (or identity ID) in the URL is a long, unguessable string, not something enforced by the rules themselves.

App Lock — sign in with mobile OTP

Adds a mobile-number + OTP screen before the app opens, using Phone Authentication on the same Firebase project as Cloud Sync above. Two things to set up first: in Firebase Console → Authentication → Sign-in method, enable Phone; and under Authentication → Settings → Authorized domains, add the domain this file is hosted on. Note: this is a convenience lock for your own device, not a secure multi-user login system — someone with direct access to this browser's storage could still bypass it, and if you're viewing this inside a sandboxed preview domain, OTP sending will likely fail until you download and host the file on your own domain.
